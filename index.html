대규모 DB 조회 성능 최적화

PostgreSQL · JSONB/TOAST · Index Only Scan · TypeORM

문제
서비스 사용량 증가로 약 2,100만 행 규모 문항 테이블에서 3초 이상 슬로우 쿼리가 반복 발생. 자료가 많은 사용자(약 15만 행 보유) 기준 조회가 16.2s, 정기 캐시 Lambda는 195.6s까지 도달.
분석
read replica에서 워스트 케이스를 EXPLAIN ANALYZE로 재현해 병목을 분리. 원인은 인덱스 부재가 아니라 JSONB 접근 비용 — 본문 HTML이 담긴 jsonb는 TOAST에 저장되고 부분 읽기가 불가능해, 메타 필드 하나를 꺼내도 행 전체를 디토스트하며 비용이 보유 행 수에 비례. 표현식 인덱스를 먼저 검증했으나 planner가 covering으로 인정하지 않아 Index Only Scan이 성립하지 않음을 확인하고 정규화 컬럼 승격을 선택. 무중단이 필수라 테이블 rewrite를 유발하는 generated column 대신 nullable 컬럼+트리거+배치 백필 조합으로 결정.
실행
정규화 컬럼·BEFORE 트리거 추가, 약 2,100만 행 keyset 배치 백필(2h 7m, 중단·재개 가능), CONCURRENTLY covering index, VACUUM ANALYZE. 별도 API의 N+1 집계 쿼리를 N회 → 1회로 재구성하고 실 DB 검증을 거친 인덱스 8종 추가.
성과
16.2s → 79ms, 4.9s → 23ms, Heap Fetches 0, 다운타임 0. 캐시 Lambda 195.6s → 35.2s, 운영 peak 17~25s → 2~5s. 결과 동치성 검증, 신규 인덱스 8종 실사용 확인.

EB 분산 운영 → ECS(EC2) 중앙 클러스터 · Terraform IaC 전환

ECS · Terraform · GitHub OIDC · ALB · Capacity Provider

문제
서비스가 늘며 EB 환경마다 Auto Scaling Group·ALB·CloudWatch가 중복 생성 — 유휴 리소스와 관리 부채가 서비스 수에 비례해 증가하는 구조.
분석
운영 서비스 트래픽 성격을 검토해 공용 ALB 공유 리스크가 낮다고 판단, 중앙 클러스터+공용 ALB+서비스별 Target Group 구조를 선택. IaC 소유권은 완전 중앙형/하이브리드/배포 레이어 분리 3안을 비교해, 통제와 서비스팀 자율성의 균형을 위해 공통 인프라는 중앙 레포가, Task Definition·Service는 서비스 레포가 소유하는 안을 채택. 전환은 Route53 Alias 스위칭으로 롤백 가능하게 설계.
실행
S3 state·DynamoDB lock·GitHub OIDC 부트스트랩, 배포/롤백 가이드 문서화, 핵심 리소스 prevent_destroy 보호. 빈 인스턴스가 축소되지 않는 이슈를 5단계로 추적해 Terraform이 ECS 관리 태그를 지우는 근본 원인을 해결.
성과
인프라 비용 구조 개선을 목표로 점진 전환 진행, 무중단 롤링 배포 약 5분, 빈 인스턴스 자동 회수 정상화, 팀원이 가이드만으로 서비스를 추가하는 운영 체계 확립.

레거시 Express API 현대화 및 백엔드 통합

Express → NestJS · Nginx 로그 분석 · Route53 컷오버

문제
로그인·인증 등 공통 기능을 담당하는 Express+Knex 레거시 서버가 주 스택(NestJS+TypeORM)과 달라 유지보수가 어렵고 전용 인스턴스 3대를 점유.
분석
스펙 문서가 없는 레거시라 코드가 아닌 실제 트래픽으로 범위를 정의 — 운영 Nginx 로그 수백만 라인을 분석해 유효 엔드포인트 52개를 추리고, 2xx/3xx 실동작 28개에서 FE 버그 호출·보안 스캔 노이즈를 제외한 27개로 축소. FE 영향 최소화를 위해 Method·Path·Response를 유지하는 API 동등성 전략을 세우고, DB 공유·ALB 라우팅·JWT/쿠키 호환·NAT egress IP·cron 의존성을 사전 검증해 롤백 가능한 컷오버로 설계.
실행
NestJS+TypeORM 재구현, 임시 도메인 검증 후 Route53 컷오버. 이후 마켓 백엔드로 흡수 통합 — 중복 엔티티 23종은 네임스페이스 분리로 무수정 공존.
성과
배포 후 12시간 ERROR 로그 60건 전수 분석으로 이상 없음 확인, 전용 인스턴스 3대 회수, 운영 ECS 서비스 1개 추가 감축.

트리거 기반 데이터 동기화 → CDC 이벤트 처리 전환

AWS DMS · Kinesis · Lambda · PostgreSQL WAL

문제
검색 테이블 동기화가 RDS 트리거 5종에 의존해 대량 업데이트 시 Lock이 전파되어 실제 서비스 다운이 발생. 소수 행 변경이 수천 건 갱신으로 파급되는 구조에 트리거 수동 on/off 부담까지 존재.
분석
DMS 기반 CDC, Logical Decoding 직접 구성, Outbox 패턴, 폴링 4안을 관리 난이도·안정성·구축 속도·확장성으로 비교. 직접 구성은 복제 슬롯 운영 부담, Outbox는 코드 침투, 폴링은 분 단위 지연이 있어 팀 규모에 맞는 관리형 DMS를 채택. 컬럼 단위 필터링 불가 한계는 before-image를 활성화해 Lambda에서 before/after 비교로 보완, 이벤트 폭증은 bulk 감지로 실시간 반영을 차단하고 배치 재처리로 우회하도록 설계.
실행
PoC → staging → prod 단계 적용, 트리거 5종 비활성화, bulk 감지+Slack 알림+배치 backfill, 복제 인스턴스는 OOM 실측으로 스펙 결정.
성과
대량 업데이트 시 Lock·장애 전파 리스크 제거, 변경 컬럼만 부분 업데이트해 의도치 않은 값 변경 방지, 전환 중 누락 데이터 복구까지 완료.

벡터 임베딩 기반 유사 지문 검색 구조 설계

pgvector · HNSW · Python Lambda · sentence-transformers

문제
외부지문 등록 시 기존 지문 수만 건과의 유사도를 검출해 중복 등록을 막아야 했고, 지문이 계속 증가하는 특성상 확장 가능한 검색 구조가 필요.
분석
기존 Doc2Vec은 지문 1건 추가마다 전체 재학습이 필요해 증가형 데이터에 부적합 — 사전학습 모델은 1건 인코딩 후 즉시 반영 가능. 모델 4종을 실제 지문으로 Top-N·유사도 점수·인코딩/검색 시간 실측 비교. BGE-M3는 품질이 가장 높았지만 2.3GB로 Lambda 메모리 한계 초과, 외부 API는 의존성·장애 리스크로 기각. e5-small-v2가 large와 품질이 거의 동일(0.947 vs 0.957)하면서 384차원 · 130MB로 비용이 낮아 채택, 판정 임계값 0.9도 실측으로 결정.
실행
pgvector+HNSW 인덱스, Python Lambda 컨테이너 embedding 서버 구성. 원본 테이블을 Source of Truth로 유지하고 변경·승인 시 즉시 갱신, 주기 fallback으로 누락 보완.
성과
brute-force 검색을 HNSW 근사 탐색 구조로 대체, 재학습 비용 제거, 모델 선정 근거를 팀 문서로 공유.

그 외 개선

  • API 레이턴시 알람 설계 — 14일 p95 실측으로 임계값을 산출하고 3일 백테스트로 알람 33건 → 11건까지 노이즈 축소, 실제 DB 장애 약 3분 내 감지 검증
  • Redis Pub/Sub 2-tier 캐시 — 오토스케일링 환경의 인스턴스 간 캐시 불일치를 L1 인메모리+L2 Redis+Pub/Sub 동기화로 해소, 장애 시나리오별 fallback 문서화
  • PDF 생성 Job 서버 분리·안정화 — API 서버에서 분리하고, 브라우저 프로세스 리소스 누수로 인한 연쇄 실패를 로그 타임라인 분석으로 규명·해결
  • 팀 개발 체계 정립 — NestJS 기술스택 선정, tag 기반 staging/prod 배포, PR 템플릿·리뷰 기준 문서화
